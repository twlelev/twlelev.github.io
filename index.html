相机参数、位姿、FOV 与 UV 反投

这份笔记整理相机内参、外参、相机位姿、FOV 的关系，以及它们在 Pixal3D 和 UV 反投中的作用。

1. 相机参数分为两类

相机参数通常分成：

内参 K：相机自身如何成像，也就是镜头和图像平面的参数
外参 R,t：相机在世界坐标中放在哪里、朝哪里看

直观理解：

内参：镜头
外参：相机架子

内参影响视野宽窄、透视强弱、图像中心位置。外参影响看到物体哪一面、相机远近、俯视还是仰视、物体在画面中的位置。

2. 内参 K

内参描述 3D 点进入相机坐标系之后，如何投影到 2D 像素坐标。常见内参矩阵：

K = [ fx   0  cx
      0   fy  cy
      0    0   1 ]

含义：

  • fx, fy：焦距，单位是像素。焦距越大，视野越窄，物体看起来越大，透视越弱。
  • cx, cy：主点，通常接近图像中心。
  • 畸变参数、skew 等在很多深度学习或 3D 生成代码里会被忽略。

如果一个点在相机坐标系下是 (Xc, Yc, Zc)，投影到像素坐标为：

u = fx * Xc / Zc + cx
v = fy * Yc / Zc + cy

这里的 (u, v) 是图像像素坐标。

3. 外参 R,t

外参描述世界坐标中的点如何变换到相机坐标。常见形式：

Xc = R * Xw + t

其中：

  • Xw：世界坐标中的 3D 点。
  • Xc：相机坐标中的 3D 点。
  • R：旋转矩阵。
  • t：平移向量。

矩阵形式：

[ Xc ]   [ R  t ] [ Xw ]
[ 1  ] = [ 0  1 ] [ 1  ]

这个 [R | t] 通常叫 world-to-camera 外参，也就是从世界坐标变到相机坐标。

4. 相机位姿 camera pose

很多图形学、SLAM、NeRF 代码里说的 camera pose，通常指的是 camera-to-world 矩阵：

Xw = R_c2w * Xc + C

矩阵形式：

T_c2w = [ R_c2w  C
          0      1 ]

其中：

  • C 是相机中心在世界坐标中的位置。
  • R_c2w 是相机坐标轴在世界坐标中的朝向。

它和 world-to-camera 外参互为逆：

R_w2c = R_c2w^T
t_w2c = -R_c2w^T * C

反过来：

R_c2w = R_w2c^T
C = -R_w2c^T * t_w2c

因此：

外参 world-to-camera = T_w2c
相机位姿 camera pose = T_c2w
T_w2c = inverse(T_c2w)

[!warning] 容易混淆
有些库把 pose 叫外参，有些库把 pose 叫外参的逆。实际使用时一定要确认矩阵方向是 world_to_camera 还是 camera_to_world。

5. 完整投影公式

世界坐标点投影到图像像素，一般分两步：

Xc = R * Xw + t

u = fx * Xc / Zc + cx
v = fy * Yc / Zc + cy

合起来可以写成：

s [u, v, 1]^T = K [R | t] [Xw, Yw, Zw, 1]^T

其中：

  • K 是内参。
  • [R | t] 是外参。
  • s 是尺度，通常对应深度。

6. 方位角、仰角、距离

在很多 3D 生成和渲染代码里，外参不直接写成 R,t，而是用更直观的参数：

azimuth   方位角：绕竖直轴转多少，例如正面、左侧、右侧
elevation 仰角：相机从上往下还是从下往上看
distance  距离：相机离物体中心多远
target    观察目标点，通常是物体中心

这些参数可以通过 look_at 方法转换成 camera pose。例如：

azimuth = 0 deg      正面
azimuth = 90 deg     侧面
elevation = 20 deg   略微俯视或仰视，取决于坐标系约定
distance = 2.5       离物体中心 2.5 个单位

得到相机位置 C 和相机朝向 R_c2w 后，取逆即可得到 world-to-camera 外参 [R | t]。

7. FOV 是什么

FOV 是 Field of View，即视场角或视野角。它描述相机镜头一次能看到多宽的范围。常见类型：

horizontal FOV / camera_angle_x：水平视场角
vertical FOV / camera_angle_y：垂直视场角
diagonal FOV：对角线视场角

FOV 属于内参相关概念。它描述镜头成像范围，不描述相机在哪里、朝哪里看。

水平 FOV 与 fx 的关系：

fov_x = 2 * atan(image_width / (2 * fx))

反过来：

fx = image_width / (2 * tan(fov_x / 2))

因此：

fx 大  -> FOV 小 -> 视野窄 -> 物体看起来更大，透视更弱
fx 小  -> FOV 大 -> 视野宽 -> 物体看起来更小，透视更强

直观例子：

长焦镜头：FOV 小，像望远镜，脸部透视被压缩
广角镜头：FOV 大，像近距离手机自拍，鼻子更突出，透视更强

如果图像宽度为 512：

fx = 512
fov_x = 2 * atan(512 / (2 * 512)) ~= 53.1 deg

fx = 256
fov_x = 2 * atan(512 / (2 * 256)) = 90 deg

这样不同视角中 person/head 的位置和画面占比才更稳定，也更适合后续 UV 反投。

8. UV 反投需要什么相机参数

如果要把多视角图像反投到 mesh 的 UV 上，仅有 MoGe 提供的内参不够。UV 反投至少需要：

1. mesh 顶点、面、UV
2. 每张图的相机内参 K：fx, fy, cx, cy
3. 每张图的相机外参 R,t 或 camera pose
4. 可见性判断：z-buffer / depth test

反投流程可以按 UV texel 做：

对每个 UV texel
  -> 找到它属于哪个 triangle
  -> 用 barycentric 插值得到 3D surface point
  -> 用 K[R|t] 投影到每张 view image
  -> depth test 判断是否可见
  -> 从图像采样颜色/PBR
  -> 多视角加权融合
  -> 对没覆盖区域做 padding / inpaint

也可以按图像像素做：

每张图像像素
  -> rasterization 得到 triangle id、barycentric、depth
  -> 找到对应 UV 坐标
  -> 把像素颜色 splat / accumulate 到 UV atlas

如果多视角图是自己从已知 mesh 渲染出来的，那么渲染时本来就有 K, R, t，此时不一定需要 MoGe。

如果多视角图是生成模型产生的，或者来自未知相机，那么 MoGe 只能提供内参线索，还必须估计每张图相对于 mesh 的外参：

R_i, t_i = camera_pose_estimation(mesh, image_i)

对于 3D head，可以使用：

  • 2D facial landmarks + FLAME/mesh 对应点做 PnP。
  • silhouette、normal、depth 的 differentiable rendering 优化。
  • face parsing、眼角、鼻尖、嘴角等语义区域约束。
  • MoGe depth/point map 与 mesh 对齐，但仍然要解相机/物体位姿。

9. Multi-view diffusion 生成图用于反投 UV

如果 multi-view images 是通过 diffusion 生成的，那么生成阶段最好明确控制或记录每张图的相机协议。至少需要：

azimuth：水平绕物体旋转角
elevation：俯仰角
distance 或 camera radius：相机离物体中心多远
target / look_at center：相机看向哪里，通常是 head center
projection type：perspective 或 orthographic
image resolution：图像宽高
coordinate convention：坐标系约定

如果使用透视投影，还需要：

FOV 或 focal length
near / far
aspect

如果使用正交投影，还需要：

ortho_scale / ortho_size
near / far
aspect

正交投影下没有透视里的 Xc / Zc 和 Yc / Zc，物体大小主要由 ortho_scale 控制。对 FLAME 或稳定拓扑 head 做 UV 反投时，正交投影常常更稳定，因为 diffusion 生成的多视角图未必有严格真实透视。

一个常见 head multi-view 协议可以是：

front:       azimuth =   0 deg, elevation = 0 deg
front-left:  azimuth = -45 deg, elevation = 0 deg
left:        azimuth = -90 deg, elevation = 0 deg
front-right: azimuth =  45 deg, elevation = 0 deg
right:       azimuth =  90 deg, elevation = 0 deg
back:        azimuth = 180 deg, elevation = 0 deg

projection = orthographic
ortho_scale = fixed
target = head center
resolution = 1024 or 2048

[!note] Camera prompt ≠ 真实相机
控制 diffusion 的 camera prompt 或 condition，不等于生成图真的严格符合该相机。比如指定 left view, azimuth=-90 deg，生成结果可能实际是 70 度侧脸，头部比例、耳朵、头发也可能发生生成式偏移。

因此反投前通常还需要做一次 pose refinement：

1. 设定目标相机 azimuth / elevation / projection
2. multi-view diffusion 生成一致视图
3. 对每张生成图重新估计或优化相机位姿、scale、translation
4. 用优化后的相机做 UV 反投

对于正交 head 视图，常优化：

azimuth
elevation
roll
translation x/y
ortho_scale

对于透视投影，还可能优化：

focal length / FOV
distance

10. nvdiffrast 中相机约定如何生效

nvdiffrast 本身不负责定义或推断相机参数。它只接收已经变换到 clip space 的顶点，以及三角面索引，然后做 rasterization 和 interpolation。典型调用类似：

rast_out, _ = dr.rasterize(ctx, pos_clip, tri, resolution=[H, W])

关键是 pos_clip：

pos_clip = MVP @ vertices_homo

其中：

MVP = Projection @ View @ Model

三部分含义：

Model：mesh 从 object/local 坐标到 world 坐标
View：world-to-camera，来自 azimuth/elevation/distance/target/look_at
Projection：camera-to-clip，来自 perspective FOV 或 orthographic scale

所以：

azimuth / elevation / distance / target
  -> 用来构造 camera pose 或 View matrix

FOV / near / far / aspect
  -> 用来构造 perspective Projection matrix

ortho_scale / near / far / aspect
  -> 用来构造 orthographic Projection matrix

resolution
  -> 传给 dr.rasterize(ctx, ..., resolution=[H, W])

crop / resize
  -> 通常不在 nvdiffrast 内部处理，需要同步修改 projection/K 或记录后处理变换

坐标系约定
  -> 体现在 View matrix、Projection matrix、顶点坐标、图像 Y 轴和 UV V 方向里

做 UV 反投时，常见流程是先用 nvdiffrast 渲染可见面，再插值得到每个像素对应的 UV：

uv_img, _ = dr.interpolate(uv, rast_out, tri)

这样每个可见像素就知道自己对应到 mesh 的哪个 UV 位置，之后就可以把该像素颜色写回 UV atlas，或者累积多视角颜色。

[!warning] nvdiffrast 中最容易出错的约定

  • 矩阵方向：如果手里是 camera-to-world pose，通常要先求逆成 world-to-camera View。
  • 相机看向方向：OpenGL 风格通常看 -Z，一些 CV 代码使用 +Z。
  • NDC 深度范围：OpenGL 常见是 [-1, 1]，一些库使用 [0, 1]。
  • 图像 Y 轴：图像数组通常 y 向下，OpenGL/NDC 常常 y 向上。
  • UV V 方向：有些 UV 约定 v=0 在底部，而图像第 0 行在顶部，常需要 v = 1 - v。
  • crop/resize：nvdiffrast 不会知道 diffusion 图后处理过，需要你把变换等效进 projection/K 或额外记录。

因此，azimuth/elevation/FOV/ortho_scale/target/坐标系 这些约定会通过你构造的 View 和 Projection 矩阵进入 nvdiffrast；nvdiffrast 只执行 rasterization 和 interpolation。

11. Crop/Resize 对内参的影响

如果图像被裁剪或缩放，内参必须同步变换，否则反投会整体偏移。

缩放 sx, sy 后：

fx' = sx * fx
fy' = sy * fy
cx' = sx * cx
cy' = sy * cy

如果先裁剪，裁剪左上角为 (x0, y0)：

cx_crop = cx - x0
cy_crop = cy - y0

再缩放到新分辨率时继续乘缩放比例。

[!important] Head reconstruction 与 UV 反投
即使内参本身估计正确，只要 crop/resize 没同步更新 K，投影和反投都会错位。

12. 总结

内参 K / FOV：描述镜头如何成像
外参 R,t：描述世界点如何变到相机坐标
相机位姿 T_c2w：描述相机在世界中的位置和朝向
world-to-camera 外参 T_w2c = inverse(T_c2w)

Pixal3D 主要用 MoGe 估计内参/FOV，用于把图像特征对齐到 3D token。它不依赖完整相机位姿。

UV 反投则必须有完整相机几何：内参、外参、mesh UV 和可见性判断。只有内参时，无法判断图像中的像素对应 mesh 的哪一面。

如果 multi-view 图来自 diffusion，生成时要明确投影类型、视角、scale/FOV、target、分辨率和坐标系约定；反投前最好再做 pose refinement。

nvdiffrast 不替用户定义相机。相机约定需要先被转换成 Model/View/Projection 矩阵，最终通过 pos_clip = MVP @ vertices_homo 进入 rasterization。

GPT-SoVITS 架构全链路详解

第一阶段: 参考音频预处理

参考音频 (3~10s)
    │
    ├─→ 16kHz resampling
    │       → CNHuBERT → HuBERT 特征 (768维, 50fps)
    │               → VQ Quantizer (RVQ, 1024码本)
    │                       → prompt_semantic  ← [1, T_ref] 离散 token
    │                                                  缓存，作为 GPT 的 prompt
    │
    └─→ 原始采样率 spectrogram → refer_spec  ← Mel 频谱，送给 SoVITS 提取音色
         (v2Pro 额外保留 16kHz 音频 → CAM++ → sv_emb 512维)

第二阶段：GPT(AR自回归)预测目标语义token

GPT的输入不只是text(音素序列 phones + BERT 特征), 还包括参考音频的语义token作为prompt


输入：
  ├─ 目标文本 → 前端处理 → 音素序列 phones + BERT 特征
  └─ prompt_semantic (参考音频的语义 token，作为上文)

GPT AR 自回归：
  [prompt_semantic | BOS] → 逐 token 预测 → pred_semantic
                                              [T_pred] 个离散 token

prompt_semantic的作用是让GPT对齐说话人的节奏/韵律风格，进行in-context仿说。

第三阶段：SoVITS解码为音频

pred_semantic (离散 token)
    → RVQ Decoder → quantized (连续向量, 768维)
    → ssl_proj + interpolate (25hz→50hz)
    ↓
TextEncoder (enc_p)：
    ├─ 输入1: quantized (语义内容)
    ├─ 输入2: phones (文本音素，控制发音)
    └─ 输入3: ge (音色/风格向量)
    → 通过 MRTE 模块融合三路信息
    → 输出: m_p, logs_p (先验分布参数)
    ↓
Flow (正规化流，逆向)：
    z_p ~ N(m_p, exp(logs_p)) → z
    ↓
Generator (HiFi-GAN 风格声码器)：
    z + ge → 波形音频

v2Pro = v2 的 SoVITS 架构 + SV Embedding + v3 的 GPT 模型，本质上是把两个最大的升级（更强语义 GPT + 说话人身份强化）合并到 v2 的轻量声学框架里，不引入外部 Vocoder，保持推理简洁

z_p、z、Flow的关系

训练：真实mel → enc_q → z ──Flow正向──→ z_p   (然后算 KL: z_p分布 vs 先验分布)
推理：先验采样 z_p ──Flow逆向──→ z → Generator
Flow：是可逆双射函数

KL loss 主要训练 enc_p（先验编码器），让它预测的 N(m_p, exp(logs_p)) 去拟合后验 z 经 Flow 正向映射后的分布。Flow 本身主要被重建损失训练。两者共同保证：推理时先验采样 z_p → Flow 逆向 → z，与训练时 enc_q(真实mel) → z 的分布一致。

模型架构速览

VibeVoice 本质上是 Qwen2.5 LLM + 扩散解码头 的混合架构，将大语言模型的文本理解能力与扩散模型的声学生成能力结合起来。

组件 作用 参数量
language_model (Qwen2.5) 自回归生成条件向量 ~1.5B
acoustic_tokenizer VAE 编解码音频波形 ~60M
semantic_tokenizer VQ-VAE 编码语义特征（自监督语音模型） ~60M
acoustic_connector 投影 acoustic latent → LLM hidden dim ~8M
semantic_connector 投影 semantic latent → LLM hidden dim ~8M
prediction_head（扩散） 噪声预测器，将 LLM hidden state → 音频 latent ~123M

文本与参考语音的融合机制

模型接受文本和参考音频两路输入，其融合发生在 embedding 层：

  1. 构造 prompt：系统提示 + [SPEECH_PLACEHOLDER × N] + Speaker 1: 目标文本 + <speech_start>
  2. Tokenize：整个 prompt 转为 token IDs
  3. 查 Embedding 表：每个 token ID 映射为向量
  4. 替换占位符：用参考音频的真实 embedding（acoustic_connector + semantic_connector 双路）覆盖占位符位置
inputs_embeds = embed_layer(input_ids).clone()
ref_speech_embeds = acoustic_connector(scaled_ac) + semantic_connector(se_latents)
ref_positions = speech_input_mask[0].nonzero()
inputs_embeds[0, ref_positions[:n_ref]] = ref_speech_embeds[0, :n_ref]

最终送入 LLM 的是 文本 embedding + 参考语音 embedding 混合序列——文本告诉它「说什么」，参考语音告诉它「用谁的声音说」。

推理中的 Semantic Feedback

自回归生成循环

每次生成一个音频 token 的流程：

for 每个输出 token:
    1. 扩散采样（DPM solver, ~10步）
       输入: last_hidden（LLM 条件向量）
       输出: scaled_latent [1, vae_dim]

    2. 反馈路由 → Acoustic-only 或 Acoustic+Semantic

    3. LLM 前向传播（单步，利用 KV cache）
       输出: 新的 last_hidden → 用于下一次扩散采样

两种反馈模式

模式 反馈内容 公式 速度
Acoustic-only 仅 acoustic embedding next = acoustic_connector(latent) 快
Acoustic+Semantic 解码→重编码→双路 next = ac_connector(latent) + se_connector(semantic_re_encode(decode(latent))) 慢（每步解码+重编码）

[!tip] 为什么 Semantic Feedback 是必要条件？
训练时 forward pass 用的是 acoustic + semantic 双路输入（有 ground truth semantic token）。Base model 推理默认用 acoustic-only——速度快，且 base model 没有严重的训练/推理分布偏移。

但微调后 prediction_head 发生变化，生成的 latent 分布与训练时不完全一致。若只反馈 acoustic，缺少 semantic 信号会让 LLM 偏离正常分布，导致退化（循环、噪声）。加上 semantic feedback 后，反馈信号与训练时一致，推理才稳定。

注意：这里的 semantic 不是 NLP 意义上「从文本得到的语义」，而是从音频波形中提取的语音内容特征（类似 HuBERT/w2vBERT）。因此必须先 decode → waveform → 再用 semantic_tokenizer 重新编码，无法从 acoustic latent 直接转换。

微调方案

方案一：Diffusion-only（主要方案）

冻结所有参数，仅训练 prediction_head（~123M params）。

原理：LLM hidden states 与 base model 完全一致，仅让扩散去噪器学习目标说话人的声学特征。

finetuning-tts/finetune_diffusion_only.py   # 训练
finetuning-tts/inference_diffusion.py       # 推理（含 semantic feedback）

方案二：LoRA on LLM（对比方案）

在 LLM 上应用 LoRA adapter（q/k/v/o/gate/up/down_proj）+ 同时训练 prediction_head。

LoRA 修改 LLM hidden states 使其更适应目标说话人，prediction_head 联合训练以适应修改后的 hidden states。

实验设计

固定参数：batch_size=1，gradient_accumulation=8（有效 batch=8），lr_scheduler=cosine，warmup_ratio=0.05，weight_decay=0.01，bf16 精度，训练数据 4704 样本。

变量 搜索范围
微调方法 diffusion-only / LoRA
学习率 (lr) 1e-7 ~ 5e-6
训练轮数 (epochs) 1 ~ 20

微调问题汇总

1.过拟合丢失生成EOS token能力

数据中 EOS：

# codec channel 中，EOS 只出现一次：在所有 audio codec 之后
input_ids[i, 8+text_ids_len-1+codec_ids_len, 1] = codec_eos_token_id

label中:

codec_0_labels[i, 8+text_ids_len-1+codec_ids_len] = codec_eos_token_id

每条训练样本中，EOS token只出现1次，而audio codec token出现几百次。
假设一条16秒音频在12HZ生成192 个codec frame, 模型看到的label分布是:

  • 192个正常codec token(如id=0~4095的各种值)
  • 1个EOS token(id=4198)

当模型在752条数据上训练20epoch出现过拟合：

  • 模型对每条样本的codec 序列背的滚瓜烂熟，loss趋于0
  • 在这种情况下，模型学习到的codec 序列和EOS绑定在一起，一起背住了
  • 当模型推理，输入没见过的text,模型找不到匹配的记忆模型，找不到EOS结束位置。

2.过拟合导致文本提示失效

loss如下:

outputs = model.talker(
    inputs_embeds=input_embeddings,  # text embedding + codec embedding 混合
    attention_mask=attention_mask,
    labels=codec_0_labels,           # 只监督 codec_0 的预测
    output_hidden_states=True
)
loss = outputs.loss + 0.3 * sub_talker_loss

loss只衡量"给定前面所有token, 预测下一个codec token的准确度"。现在考虑两种降低loss的策略:

  1. (理想泛化学习)理解文本内容 -> 根据文本语义发音 -> 预测对应的codec token
  2. (死记硬背)根据codec token序列[a,b,c,…] -> 直接记住a后面是b,b后面是c,预测下一个codec token

两种策略都能然loss下降，但策略2更简单，梯度下降优先选择这条路。

  1. 输入中 codec embedding 和 text embedding 是叠加的
input_embeddings = input_text_embedding + input_codec_embedding

模型可以选择性关注codec通道而忽略text通道
2. codec自身具有极强的自相关性 - 语音信号是连续的，相邻帧的codec是高度相似的。仅靠前几帧就能预测下一帧，不需要看文本。

3.过拟合导致重复循环

训练时：不存在重复循环
训练时用的是teacher forcing， 上下文中每一个codec token都是ground truth.所以训练时模型永远看到正确的上下文，不会出现循环。Loss 下降是正常的。
推理时
推理用的是autoregressive decoding - 每一步的输入是模型上一步生成的token.
这就是 train-inference mismatch（训练-推理不一致）问题,推理时很快偏离真实分布。

过拟合的模型在推理时会这样崩溃：

  1. 第 1 步：输入新文本，模型没见过，输出一个不太对的 codec token
  2. 第 2 步：输入 X，这个 X 不在任何训练样本的正常上下文中，模型又输出一个偏离的 Y
  3. 误差累积：几步之后，模型进入一个它在训练中完全没见过的状态空间
  4. 在陌生状态下：模型的输出概率分布变得混乱，很容易产生一个短序列 [A, B, C] 使得 A→B, B→C, C→A 形成闭环
  5. 闭环一旦形成就无法逃脱 — 因为模型没有"识别自己在循环"的能力，且不会生成 EOS

HeliosPipeline vs 2D 图像生成 Pipeline 的核心区别

HeliosPipeline（视频生成）与标准 2D 图像生成（如 StableDiffusion / FLUX）的主要差异：

1. Latents 维度：5D vs 4D

对比项 2D 图像生成 Helios 视频生成
Latent shape (B, C, H, W) 4维 (B, C, T, H, W) 5维
示例 (1, 4, 64, 64) (1, 16, 9, 48, 80)

代码依据（prepare_latents）：

num_latent_frames = (num_frames - 1) // self.vae_scale_factor_temporal + 1
shape = (
    batch_size,
    num_channels_latents,  # 16 channels
    num_latent_frames,     # 时间维度 T
    int(height) // self.vae_scale_factor_spatial,   # H/8
    int(width) // self.vae_scale_factor_spatial,    # W/8
)

2. VAE：时间 + 空间双重下采样

对比项 2D 图像 VAE Helios VAE（AutoencoderKLWan）
空间下采样 spatial_scale = 8 vae_scale_factor_spatial = 8
时间下采样 无 vae_scale_factor_temporal = 4
通道数 通常 4 in_channels = 16

时间压缩公式：

num_latent_frames = (num_frames - 1) // 4 + 1
# 例如：132帧视频 → (132-1)//4 + 1 = 33 个 latent frames

3. 分块自回归生成（Chunk-based Autoregressive）

这是 Helios 与 2D 图像生成最大的设计差异。

  • 2D 图像：一次 denoise 整张图，无历史信息
  • Helios：视频按 chunk 逐段生成，每个 chunk 为 window_num_frames 帧，并维护多尺度历史 latent 缓冲区

历史帧的注入机制：Concat to Token Sequence，而非 KV Cache

源码:transformer_helios_diffusers.py，历史帧不是通过 KV Cache 注入，而是通过将历史 latent 编码成 token，直接 concat 到当前 chunk 的 token sequence 前面，一起参与 Self-Attention。

核心机制：Token Concat Before Self-Attention

在 HeliosTransformer3DModel.forward() 中，流程如下：

Step 1：当前 chunk noisy latent patchify

hidden_states = self.patch_embedding(hidden_states)  
# shape: (B, inner_dim, T_cur, H', W')
hidden_states = hidden_states.flatten(2).transpose(1, 2)
# shape: (B, T_cur*H'*W', inner_dim)
original_context_length = hidden_states.shape[1]  # 记录当前chunk的token数

Step 2：Short 历史 latent 用独立 conv 编码后 concat 到前面

latents_history_short = self.patch_short(latents_history_short)  # 1x patch
# flatten → (B, T_short*H'*W', inner_dim)
hidden_states = torch.cat([latents_history_short, hidden_states], dim=1)
rotary_emb = torch.cat([rotary_emb_history_short, rotary_emb], dim=1)

Step 3：Mid 历史 latent 用2x 下采样 conv 编码后 concat

latents_history_mid = self.patch_mid(latents_history_mid)  # 2x patch（token数变少）
hidden_states = torch.cat([latents_history_mid, hidden_states], dim=1)

Step 4：Long 历史 latent 用4x 下采样 conv 编码后 concat

latents_history_long = self.patch_long(latents_history_long)  # 4x patch（token数更少）
hidden_states = torch.cat([latents_history_long, hidden_states], dim=1)

最终进入 Transformer Block 的 token 序列结构：

[ long_history_tokens | mid_history_tokens | short_history_tokens | current_chunk_tokens ]
                           ↑ 全部参与 Self-Attention ↑

与 KV Cache 的本质区别

特性 KV Cache（如 LLM） Helios 的做法
历史信息存储 存储历史 K/V，复用 存储历史 latent，每次重新编码
注入方式 历史 K/V 与当前 Q 做 attention 历史 token 与当前 token concat 后共同做 Self-Attention
Q 的范围 只有当前 token 做 Q 历史+当前 token 全部做 Q（但 output 只取 current 部分）
信息流向 单向：历史→当前 双向全 attention（但历史帧用 t=0 timestep）

两个精妙的设计细节

1. original_context_length：output 只取当前 chunk 部分

# HeliosOutputNorm.forward()
hidden_states = hidden_states[:, -original_context_length:, :]  # 只取当前 chunk 的 token 输出

历史 token 参与 attention 贡献 context，但最终 output 只对当前 chunk 的 token 解码，历史 token 的 output 被丢弃。

2. 历史帧使用 t=0 的 timestep embedding（已经干净的帧）

if self.zero_history_timestep:
    timestep_t0 = torch.zeros((1), dtype=timestep.dtype, ...)
    temb_t0, timestep_proj_t0, _ = self.condition_embedder(timestep_t0, ...)
    # 历史 token 使用 t=0 的 timestep embedding
    temb = torch.cat([temb_t0, temb], dim=1)

这在语义上非常合理：历史帧是已经生成好的"干净"帧，不需要去噪，所以 timestep=0。

3. 多尺度历史：空间&时间分辨率递减

short_history: patch_size = (1,2,2)  → 原始分辨率，距当前最近 1帧
mid_history:   patch_size = (2,4,4)  → 空间 1/2，时间 1/2，中期 2帧
long_history:  patch_size = (4,8,8)  → 空间 1/4，时间 1/4，远期 16帧

时间越久的历史，其 token 数越少（分辨率越低），这样既保留了长程依赖，又控制了 sequence length 的增长—— 类似于人类记忆的多分辨率压缩。

总结

Helios 历史帧的注入方式是：Multi-scale Token Concatenation（多尺度历史 token 拼接）到 Self-Attention 的 token 序列前。

  • ✅ 不是 KV Cache（没有缓存 K/V，每次重新编码历史 latent）。
  • ✅ 不是替换 T 中的部分帧（当前 chunk 的 T 维度是完整的 noisy latent，历史从 token 维度拼接进来）。
  • ✅ Self-Attention 可以看到历史 token（全局注意力），但不消耗于 output。

4.1 CA 引擎的工作原理：刻录决策路径

总结：

1. 标准注意力机制的问题：

标准机制需要“频繁读写”，它在GPU上的计算逻辑如下：

  • 任务 A (MatMul)：从 HBM 读 Q, K →\rightarrow→ 算出 SSS →\rightarrow→ 写回 HBM（因为 SRAM 放不下整个 N×NN \times NN×N）。
  • 任务 B (Softmax)： 从 HBM 读回 SSS →\rightarrow→ 处理 →\rightarrow→ 写回 PPP 到 HBM。
  • 任务 C (MatMul)：从 HBM 读回 P,VP, VP,V →\rightarrow→ 算出 OOO →\rightarrow→ 写回 HBM。

2.FlashAttention解决思路

SRAM (片上高性能内存）较小，gpu kernel计算从SRAM 中读取数据，SRAM 相比较于HBM没有IO瓶颈(IO读写速度高一个数量级)，但是内存小。因此flashattention利用这一点，利用tiling计算将所有中间结果都留在SRAM，既降低了显存又减少了SRAM与HBM之间的IO,当得到最终结果再放回HBM。

FlashAttention 的核心思想是 “计算换 IO”。它主要通过以下两个核心技术实现：

  • 分块 (Tiling)： 它将输入的查询（Query）、键（Key）和值（Value）切分成小块，加载到 SRAM（速度极快但容量小）中。在 SRAM 里完成局部的注意力计算（包括 Softmax 的部分求和）。
  • 重计算/在线 Softmax (Recomputation / Online Softmax)： 标准的 Softmax 需要全局信息。FlashAttention 使用了一种数学技巧（Online Softmax），允许它在不存储完整得分矩阵的情况下，通过不断更新局部最大值和求和项，最终计算出精确的注意力输出。在反向传播（训练时）中，它也不存储中间矩阵，而是通过存储的统计量在需要时重新计算。

技术深度解析：LCM-LoRA 原理与 Scheduler 源码实现

1. 背景介绍

在扩散模型（Diffusion Models）的推理过程中，传统的采样方法（如 DDIM, Euler）通常需要 20-50 步迭代。一致性模型（Consistency Models, CM） 的出现旨在打破这一限制，通过学习一个“一致性映射”，模型可以直接从噪声状态 xtx_txt​ 预测到 x0x_0x0​。
本质： 不是在求解 ODE，而是在直接预测 ODE 轨迹的不动点。

LCM-LoRA 则是将 LCM 的蒸馏技术与 LoRA（低秩自适应）结合，使得原本庞大的采样步数可以压缩至 2-8 步，且可以像普通 LoRA 一样灵活插拔，极大地降低了推理成本。

2. 核心数学理论

2.1 概率流 ODE (PF-ODE)

扩散模型的前向过程可以映射为一个概率流常微分方程（Probability Flow ODE）。传统的采样器本质上是在数值求解这个 ODE。

2.2 一致性映射（Consistency Mapping）

一致性模型旨在学习一个函数 f(xt,t)f(x_t, t)f(xt​,t)，其满足：
f(xt,t)=f(xt′,t′)=x0f(x_t, t) = f(x_{t'}, t') = x_0f(xt​,t)=f(xt′​,t′)=x0​
这意味着对于同一条 PF-ODE 轨迹上的任何点，无论时间步 ttt 是多少，经过该函数映射后都应该指向同一个终点 x0x_0x0​。

# 直接计算下一状态 (LCM 模型的一致性步进)
# prev_sample = x_{t-1}
prev_sample = c_skip * sample + c_out * pred_original_sample

if self.num_inference_steps > 1 and timestep > 0:
    noise = randn_tensor(model_output.shape, generator=generator)
