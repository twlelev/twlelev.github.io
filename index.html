Part A · codebase-memory-mcp

仓库：DeusData/codebase-memory-mcp

一、它是什么

一个纯 C 写、零依赖、单二进制的 MCP 服务器：用 tree-sitter 把代码库解析成持久化知识图谱，让 AI agent 用一条结构化图查询替代几十轮 grep/read，把"理解大型代码库"的 token 开销从约 41 万压到约 3,400。

核心价值：

痛点 它的解法
agent 读代码靠 grep + 整文件塞上下文，token 爆炸 一次建图，之后"谁调谁/谁实现谁/改这里炸哪里"都是单次图查询，只返回符号关系不返回源码
grep 是文本层操作，不懂"调用/实现/导入"结构语义 13 种节点 + 20+ 关系的知识图谱，openCypher 查询亚毫秒返回
传统图数据库磁盘 IO 慢 全程内存 SQLite + LZ4，最后单次落盘；Linux 内核 2800 万行 3 分钟建图

和 graphify 的分工：codebase-memory-mcp 只吃代码、极致优化 token 效率；graphify 吃代码/文档/论文/图片、产出社区聚类 wiki。研究语料库用 graphify；接入真实大型代码仓库时才考虑 codebase-memory-mcp。

二、安装

# 一键安装（自动探测并配置已安装的 agent）—— 注意用 bash 不是 sh

⚠️ 常见坑：| sh 报 set: Illegal option -o pipefail
Debian/Ubuntu 的 /bin/sh 是 dash，不认 bash 专属的 set -o pipefail。脚本本身没问题，是解释器选错了。把 | sh 换成 | bash 即可。

less /tmp/cbm-install.sh        # 扫一眼装了啥、装哪

极简容器若没装 bash：which bash 查不到就先 apt-get install -y bash。

  • 安装时会自动探测并写入最多 11 个 agent 的配置：Claude Code（含 .mcp.json + 4 个 Skills + Grep/Glob 的 PreToolUse hook）、Codex CLI、Gemini CLI、Zed、VS Code、Aider 等。

四、核心用法：建图 → 查询

1. 索引一个仓库（每个新项目首次都要做一次）

方式一：让 Claude 调用 index_repository 工具，或直接说：

帮我索引 /path/to/repo 这个代码库

方式二：命令行直接调二进制

codebase-memory-mcp index /path/to/repo
  • 建图是一次性的；之后图谱持久化在 ~/.cache/，跨会话存活，不需要每次进项目重新索引。
  • 增量自动同步：工具内置 git-polling，检测到代码改动会自动更新图谱。

2. 如何触发 Claude 用图谱而非 grep

三种方式，从被动到主动：

  1. PreToolUse hook（自动、透明）：装好后，Claude 想 grep/glob 时，hook 自动用图谱增强结果——你什么都不用做。
  2. 显式要求：在 prompt 里说"用 codebase-memory 查/用知识图谱查调用链"，Claude 会优先调 MCP 工具。
  3. 直接给结构化问题：问"main 最终调到哪些数据库写操作""改 X 会影响哪些函数"这类结构关系问题，Claude 自然倾向用 trace_path/query_graph。

3. 常用 Cypher 查询示例

-- 1) main 5 层内调用到的所有写操作
MATCH (f:Function)-[:CALLS*1..5]->(w) WHERE f.name = 'main' AND w.label = 'WRITES' RETURN w.name

-- 2) 死代码检测：没有任何调用者的函数
MATCH (f:Function) WHERE NOT EXISTS { (f)<-[:CALLS]-() } RETURN f.name

-- 3) 谁实现了某接口
MATCH (c:Class)-[:IMPLEMENTS]->(i:Interface) WHERE i.name = 'Handler' RETURN c.name

查询引擎是只读 openCypher 子集：支持 MATCH/WHERE/WITH/RETURN/ORDER BY/聚合等；不支持写操作（它是"理解层"不是"编辑层"），遇到不支持的语法会抛清晰错误而非静默返回空。

六、多分支怎么办

工具按 git 当前 checkout 的状态建图，没有内置 per-branch 隔离——图谱按仓库路径存储，只反映"这个目录此刻磁盘上是什么代码"。

切分支后，git-polling 会发现 HEAD 变了并自动把那唯一一份图谱同步成新分支的状态（你不用手动重 index），但它是"覆盖式变形"不是"新增一份"，旧分支的图谱同时就没了。另外 polling 是轮询不是实时，切分支后到图谱更新有延迟（检测延迟 + 增量重建），想立即刷新就手动 codebase-memory-mcp index <路径> 催一次，或用 index-status 看图谱当前对应哪个 commit。

想同时保留多个分支的图谱，用 worktree 分隔：

git worktree add ../repo-featureX featureX     # 每个分支一个独立工作目录
# 在每个 worktree 里各 index 一次 → 各自独立图谱，互不污染

因为是两个不同路径，就成了两份独立图谱，路径名本身还能充当分支标签，永远不会困惑"这图是哪个分支的"。

七、适用边界（什么时候别用）

适合：大型 repo（10 万行+）接 Claude Code 省 token；跨文件架构/调用链分析；改动爆炸半径评估；死代码检测；Python/TS/Go/Rust/Java/C# 等 9 种主流语言。

要小心：

  • 聚合查询大库会少计（200 行查询上限，COUNT 静默截断）——别拿它做精确统计。
  • 属性提取不全——很多场景"返回了函数但元数据是空"。
  • 长尾语言（OCaml/Haskell/Scala/Ruby 抽象方法）精度明显下降。
  • 纯静态分析天花板——动态分发/反射/运行时元编程的调用边追不全（ingest_traces 部分缓解）。
  • 只读——它是理解层，不能用它改图改代码。

1. 它是什么

把任意一个文件夹（论文 + 推文 + 截图 + 代码 + 笔记都行）变成可导航的知识图谱，三件 Claude 单独做不到的事：

能力 说明
持久化图谱 关系存在 graphify-out/graph.json，跨会话存活，几周后还能直接查，不用重读所有文件
诚实审计轨迹 每条边标 EXTRACTED（源里明确）/ INFERRED（合理推断）/ AMBIGUOUS（存疑待审），你知道哪些是"找到的"哪些是"推的"
跨文档惊喜 社区检测（Louvain）找出不同文件里你根本想不到去问的概念关联（god nodes、surprising connections、hyperedges）

核心差异 vs codebase-memory-mcp：codebase-memory 只吃代码、只画结构图（调用/实现/导入）、极致省 token；graphify 吃一切输入、产出社区聚类 + wiki + 人话报告，是"知识图谱"而非纯"代码图谱"。

2. 双层提取（决定了成本）

层 处理对象 机制 成本
结构层（AST） 代码文件 确定性 tree-sitter 解析 免费、自动
语义层 文档/论文/图片 派发 Claude 子 agent 抽取概念/引用/图意 花 token

关键推论：改代码可以免费自动更新图；改文档/论文/图片要手动 /graphify --update（语义抽取烧 token）。如果你的语料库主要是论文/文档，它不会自动更新，得记着手动跑。

3. 常用命令（在 Claude Code 对话里输入）

/graphify                          # 对当前目录跑完整管线 → Obsidian vault
/graphify <path>                   # 对指定路径跑
/graphify <path> --update          # 增量：只重新抽取新增/改动的文件（最常用）
/graphify <path> --mode deep       # 更激进的 INFERRED 边，更深抽取
/graphify <path> --cluster-only    # 只在现有图上重跑社区聚类
/graphify <path> --no-viz          # 跳过可视化，只出报告 + JSON

/graphify add <url>                # 抓网页存进 ./raw 并更新图
/graphify query "<问题>"            # BFS 遍历，给宽上下文回答
/graphify query "<问题>" --dfs      # DFS，追一条具体路径
/graphify path "A" "B"             # 求两个概念间最短路径
/graphify explain "SwinTransformer"# 用人话解释某个节点

导出格式（按需加 flag）：--svg（贴 Notion/GitHub）、--graphml（Gephi/yEd）、--neo4j（生成 cypher.txt）、--mcp（起 MCP server 给 agent 访问）。

4. 产出物（都在 graphify-out/）

  • GRAPH_REPORT.md — 人话报告：god nodes（核心抽象）、社区结构、surprising connections、建议追问。回答架构/全局问题前先读它。
  • graph.json — GraphRAG-ready 的机器可读图。
  • graph.html — 交互式可视化（注意：节点 > 5000 时自动跳过 HTML，需用 Obsidian vault 或 --svg 看）。
  • wiki/index.md（若生成）— 导航优先看它，别翻原始文件。

5. 更新行为

  • 代码改动：python3 -c "from graphify.watch import _rebuild_code; from pathlib import Path; _rebuild_code(Path('.'))" 免费重建。
  • 文档/论文/图片改动：手动 /graphify <path> --update，语义抽取花 token。
  • 建议在项目的 CLAUDE.md 里记一行"上次全量更新时间戳 + 基线统计"，超 ~30 天没更新就提醒重跑——文档型语料库不会自动更新，很容易忘。

FLAME

![][/images/3D/Pasted_image_20260622135821.png]

核心：屏幕空间稠密先验(screen-space dense priors)。

“search uv” 机制详解

“search uv” 的本质是:用 KNN 在网络预测的 UV 图里,为每个 FLAME 顶点反查出它对应的图像像素位置,从而把"per-pixel 的 UV 预测"转成"per-vertex 的 2D 重投影目标"来拟合 FLAME 参数。

个人总结，通俗解释：
网络预测uv coordinates, 得到屏幕像素空间与uv坐标的映射，因为canonical uv坐标与3d顶点存在对应的关系，因此通过转换可以得到屏幕像素空间与3d顶点的对应。
预测的uv坐标不一定准确，所以通过search_uv优化，将其与canonical uv比较(最邻近搜索，排除距离远的奇异点等)得到屏幕空间每一个像素对应的3d坐标作为gt来监督。

A.1 前提:两套 UV 的定义

(a) Canonical UV(模板顶点的固定 UV) — losses.py:60-61

self.can_uv = torch.from_numpy(np.load(env_paths.FLAME_UV_COORDS)[self.valid_verts, :]).cuda().unsqueeze(0).float()
self.can_uv[..., 1] = (self.can_uv[..., 1] * -1) + 1   # OBJ-v 翻成 image-style v
  • FLAME_UV_COORDS 是 FLAME 模板每个顶点在 UV 图集上的固定坐标,与姿态/表情无关(对应正文 §5 的"横向独立平面 UV ∈ [0,1]²")。
  • valid_verts(losses.py:56-59)剔除后脑勺/脖子等不可靠区域;stricter_mask 时用更窄的集合。
  • 第 61 行 v = 1 - v,把 OBJ 约定翻成图像坐标,与网络预测 UV 同约定。

(b) Predicted UV(网络对每个图像像素的 UV 预测) — 即 uv_map,网络输出 2 通道,每像素值是"该像素落在 FLAME UV 图集的哪个位置"。

A.2 核心搜索:compute_corresp

losses.py:79-95:


gt_uv = gt[:, :2, :, :].permute(0, 2, 3, 1)           # 取预测 UV 的 2 通道
gt_uv = gt_uv.reshape(gt_uv.shape[0], -1, 2)          # B x (H*W) x 2,每个像素当成 UV 空间一个点
can_uv = self.can_uv.repeat(gt_uv.shape[0], 1, 1)     # B x N_verts x 2
knn_result = knn_points(can_uv, gt_uv)                # 关键:在 UV 空间做最近邻
pixel_position_width  = knn_result.idx % gt.shape[-1] # 展平索引拆回 (x, y)
pixel_position_height = knn_result.idx // gt.shape[-2]
self.dists = knn_result.dists.clone()
self.gt_2_verts = torch.cat([pixel_position_width, pixel_position_height], dim=-1)

要点:

  • 搜索发生在 UV 空间,而非图像空间。 knn_points(can_uv, gt_uv) 对每个顶点的 canonical UV,在全部 H*W 个像素的预测 UV 里找欧氏距离最近的像素。含义:“哪个图像像素的预测 UV,最接近这个顶点本该所在的 UV?”
  • knn_result.idx 是展平像素下标,% width / // width 还原成 (x, y) → 存进 gt_2_verts(每个有效顶点一个 2D 目标点)。
  • dists 记录每个匹配的 UV 距离,用作置信度门控。
  • @torch.compiler.disable(losses.py:79):KNN 不走 torch.compile。
  • GT 侧 UV 在 uv_pred_to_mesh 里(uv.py:51-55)先 (x+1)/2 归一化、用 mask 清背景、把 abs>=1 的无效预测设成 10(推到 UV 无穷远), 保证无效像素永不被选中。

A.3 把搜索结果变成 loss:compute_loss

losses.py:99-127:

valid_proj_v = proj_vertices[:, self.valid_vertex_index, ..., :2]  # 当前 FLAME 参数投影出的顶点 2D
v_dist_2d = (gt_2_verts - valid_proj_v)                            # 目标像素 - 投影像素
uv_loss = (

reconstruct中相机相关基础

2026-05-18
基础

相机参数、位姿、FOV 与 UV 反投

这份笔记整理相机内参、外参、相机位姿、FOV 的关系，以及它们在 Pixal3D 和 UV 反投中的作用。

1. 相机参数分为两类

相机参数通常分成：

内参 K：相机自身如何成像，也就是镜头和图像平面的参数
外参 R,t：相机在世界坐标中放在哪里、朝哪里看

直观理解：

内参：镜头
外参：相机架子

内参影响视野宽窄、透视强弱、图像中心位置。外参影响看到物体哪一面、相机远近、俯视还是仰视、物体在画面中的位置。

2. 内参 K

内参描述 3D 点进入相机坐标系之后，如何投影到 2D 像素坐标。常见内参矩阵：

K = [ fx   0  cx
      0   fy  cy
      0    0   1 ]

含义：

  • fx, fy：焦距，单位是像素。焦距越大，视野越窄，物体看起来越大，透视越弱。
  • cx, cy：主点，通常接近图像中心。
  • 畸变参数、skew 等在很多深度学习或 3D 生成代码里会被忽略。

如果一个点在相机坐标系下是 (Xc, Yc, Zc)，投影到像素坐标为：

u = fx * Xc / Zc + cx
v = fy * Yc / Zc + cy

这里的 (u, v) 是图像像素坐标。

3. 外参 R,t

外参描述世界坐标中的点如何变换到相机坐标。常见形式：

Xc = R * Xw + t

其中：

  • Xw：世界坐标中的 3D 点。
  • Xc：相机坐标中的 3D 点。
  • R：旋转矩阵。
  • t：平移向量。

矩阵形式：

[ Xc ]   [ R  t ] [ Xw ]
[ 1  ] = [ 0  1 ] [ 1  ]

这个 [R | t] 通常叫 world-to-camera 外参，也就是从世界坐标变到相机坐标。

4. 相机位姿 camera pose

很多图形学、SLAM、NeRF 代码里说的 camera pose，通常指的是 camera-to-world 矩阵：

Xw = R_c2w * Xc + C

矩阵形式：

T_c2w = [ R_c2w  C
          0      1 ]

其中：

  • C 是相机中心在世界坐标中的位置。
  • R_c2w 是相机坐标轴在世界坐标中的朝向。

它和 world-to-camera 外参互为逆：

R_w2c = R_c2w^T
t_w2c = -R_c2w^T * C

反过来：

R_c2w = R_w2c^T
C = -R_w2c^T * t_w2c

因此：

外参 world-to-camera = T_w2c
相机位姿 camera pose = T_c2w
T_w2c = inverse(T_c2w)

[!warning] 容易混淆
有些库把 pose 叫外参，有些库把 pose 叫外参的逆。实际使用时一定要确认矩阵方向是 world_to_camera 还是 camera_to_world。

5. 完整投影公式

世界坐标点投影到图像像素，一般分两步：

Xc = R * Xw + t

u = fx * Xc / Zc + cx
v = fy * Yc / Zc + cy

合起来可以写成：

s [u, v, 1]^T = K [R | t] [Xw, Yw, Zw, 1]^T

其中：

  • K 是内参。
  • [R | t] 是外参。
  • s 是尺度，通常对应深度。

6. 方位角、仰角、距离

在很多 3D 生成和渲染代码里，外参不直接写成 R,t，而是用更直观的参数：

azimuth   方位角：绕竖直轴转多少，例如正面、左侧、右侧
elevation 仰角：相机从上往下还是从下往上看
distance  距离：相机离物体中心多远
target    观察目标点，通常是物体中心

这些参数可以通过 look_at 方法转换成 camera pose。例如：

azimuth = 0 deg      正面
azimuth = 90 deg     侧面
elevation = 20 deg   略微俯视或仰视，取决于坐标系约定
distance = 2.5       离物体中心 2.5 个单位

得到相机位置 C 和相机朝向 R_c2w 后，取逆即可得到 world-to-camera 外参 [R | t]。

7. FOV 是什么

FOV 是 Field of View，即视场角或视野角。它描述相机镜头一次能看到多宽的范围。常见类型：

horizontal FOV / camera_angle_x：水平视场角
vertical FOV / camera_angle_y：垂直视场角
diagonal FOV：对角线视场角

FOV 属于内参相关概念。它描述镜头成像范围，不描述相机在哪里、朝哪里看。

水平 FOV 与 fx 的关系：

fov_x = 2 * atan(image_width / (2 * fx))

反过来：

fx = image_width / (2 * tan(fov_x / 2))

因此：

fx 大  -> FOV 小 -> 视野窄 -> 物体看起来更大，透视更弱
fx 小  -> FOV 大 -> 视野宽 -> 物体看起来更小，透视更强

直观例子：

长焦镜头：FOV 小，像望远镜，脸部透视被压缩
广角镜头：FOV 大，像近距离手机自拍，鼻子更突出，透视更强

如果图像宽度为 512：

fx = 512
fov_x = 2 * atan(512 / (2 * 512)) ~= 53.1 deg

fx = 256
fov_x = 2 * atan(512 / (2 * 256)) = 90 deg

这样不同视角中 person/head 的位置和画面占比才更稳定，也更适合后续 UV 反投。

8. UV 反投需要什么相机参数

如果要把多视角图像反投到 mesh 的 UV 上，仅有 MoGe 提供的内参不够。UV 反投至少需要：

1. mesh 顶点、面、UV
2. 每张图的相机内参 K：fx, fy, cx, cy
3. 每张图的相机外参 R,t 或 camera pose
4. 可见性判断：z-buffer / depth test

反投流程可以按 UV texel 做：

对每个 UV texel
  -> 找到它属于哪个 triangle
  -> 用 barycentric 插值得到 3D surface point
  -> 用 K[R|t] 投影到每张 view image
  -> depth test 判断是否可见
  -> 从图像采样颜色/PBR
  -> 多视角加权融合
  -> 对没覆盖区域做 padding / inpaint

也可以按图像像素做：

每张图像像素
  -> rasterization 得到 triangle id、barycentric、depth
  -> 找到对应 UV 坐标
  -> 把像素颜色 splat / accumulate 到 UV atlas

如果多视角图是自己从已知 mesh 渲染出来的，那么渲染时本来就有 K, R, t，此时不一定需要 MoGe。

如果多视角图是生成模型产生的，或者来自未知相机，那么 MoGe 只能提供内参线索，还必须估计每张图相对于 mesh 的外参：

R_i, t_i = camera_pose_estimation(mesh, image_i)

对于 3D head，可以使用：

  • 2D facial landmarks + FLAME/mesh 对应点做 PnP。
  • silhouette、normal、depth 的 differentiable rendering 优化。
  • face parsing、眼角、鼻尖、嘴角等语义区域约束。
  • MoGe depth/point map 与 mesh 对齐，但仍然要解相机/物体位姿。

9. Multi-view diffusion 生成图用于反投 UV

如果 multi-view images 是通过 diffusion 生成的，那么生成阶段最好明确控制或记录每张图的相机协议。至少需要：

azimuth：水平绕物体旋转角
elevation：俯仰角
distance 或 camera radius：相机离物体中心多远
target / look_at center：相机看向哪里，通常是 head center
projection type：perspective 或 orthographic
image resolution：图像宽高
coordinate convention：坐标系约定

如果使用透视投影，还需要：

FOV 或 focal length
near / far
aspect

如果使用正交投影，还需要：

ortho_scale / ortho_size
near / far
aspect

正交投影下没有透视里的 Xc / Zc 和 Yc / Zc，物体大小主要由 ortho_scale 控制。对 FLAME 或稳定拓扑 head 做 UV 反投时，正交投影常常更稳定，因为 diffusion 生成的多视角图未必有严格真实透视。

一个常见 head multi-view 协议可以是：

front:       azimuth =   0 deg, elevation = 0 deg
front-left:  azimuth = -45 deg, elevation = 0 deg
left:        azimuth = -90 deg, elevation = 0 deg
front-right: azimuth =  45 deg, elevation = 0 deg
right:       azimuth =  90 deg, elevation = 0 deg
back:        azimuth = 180 deg, elevation = 0 deg

projection = orthographic
ortho_scale = fixed
target = head center
resolution = 1024 or 2048

[!note] Camera prompt ≠ 真实相机
控制 diffusion 的 camera prompt 或 condition，不等于生成图真的严格符合该相机。比如指定 left view, azimuth=-90 deg，生成结果可能实际是 70 度侧脸，头部比例、耳朵、头发也可能发生生成式偏移。

因此反投前通常还需要做一次 pose refinement：

1. 设定目标相机 azimuth / elevation / projection
2. multi-view diffusion 生成一致视图
3. 对每张生成图重新估计或优化相机位姿、scale、translation
4. 用优化后的相机做 UV 反投

对于正交 head 视图，常优化：

azimuth
elevation
roll
translation x/y
ortho_scale

对于透视投影，还可能优化：

focal length / FOV
distance

10. nvdiffrast 中相机约定如何生效

nvdiffrast 本身不负责定义或推断相机参数。它只接收已经变换到 clip space 的顶点，以及三角面索引，然后做 rasterization 和 interpolation。典型调用类似：

rast_out, _ = dr.rasterize(ctx, pos_clip, tri, resolution=[H, W])

关键是 pos_clip：

pos_clip = MVP @ vertices_homo

其中：

MVP = Projection @ View @ Model

三部分含义：

Model：mesh 从 object/local 坐标到 world 坐标
View：world-to-camera，来自 azimuth/elevation/distance/target/look_at
Projection：camera-to-clip，来自 perspective FOV 或 orthographic scale

所以：

azimuth / elevation / distance / target
  -> 用来构造 camera pose 或 View matrix

FOV / near / far / aspect
  -> 用来构造 perspective Projection matrix

ortho_scale / near / far / aspect
  -> 用来构造 orthographic Projection matrix

resolution
  -> 传给 dr.rasterize(ctx, ..., resolution=[H, W])

crop / resize
  -> 通常不在 nvdiffrast 内部处理，需要同步修改 projection/K 或记录后处理变换

坐标系约定
  -> 体现在 View matrix、Projection matrix、顶点坐标、图像 Y 轴和 UV V 方向里

做 UV 反投时，常见流程是先用 nvdiffrast 渲染可见面，再插值得到每个像素对应的 UV：

uv_img, _ = dr.interpolate(uv, rast_out, tri)

这样每个可见像素就知道自己对应到 mesh 的哪个 UV 位置，之后就可以把该像素颜色写回 UV atlas，或者累积多视角颜色。

[!warning] nvdiffrast 中最容易出错的约定

  • 矩阵方向：如果手里是 camera-to-world pose，通常要先求逆成 world-to-camera View。
  • 相机看向方向：OpenGL 风格通常看 -Z，一些 CV 代码使用 +Z。
  • NDC 深度范围：OpenGL 常见是 [-1, 1]，一些库使用 [0, 1]。
  • 图像 Y 轴：图像数组通常 y 向下，OpenGL/NDC 常常 y 向上。
  • UV V 方向：有些 UV 约定 v=0 在底部，而图像第 0 行在顶部，常需要 v = 1 - v。
  • crop/resize：nvdiffrast 不会知道 diffusion 图后处理过，需要你把变换等效进 projection/K 或额外记录。

因此，azimuth/elevation/FOV/ortho_scale/target/坐标系 这些约定会通过你构造的 View 和 Projection 矩阵进入 nvdiffrast；nvdiffrast 只执行 rasterization 和 interpolation。

11. Crop/Resize 对内参的影响

如果图像被裁剪或缩放，内参必须同步变换，否则反投会整体偏移。

缩放 sx, sy 后：

fx' = sx * fx
fy' = sy * fy
cx' = sx * cx
cy' = sy * cy

如果先裁剪，裁剪左上角为 (x0, y0)：

cx_crop = cx - x0
cy_crop = cy - y0

再缩放到新分辨率时继续乘缩放比例。

[!important] Head reconstruction 与 UV 反投
即使内参本身估计正确，只要 crop/resize 没同步更新 K，投影和反投都会错位。

12. 总结

内参 K / FOV：描述镜头如何成像
外参 R,t：描述世界点如何变到相机坐标
相机位姿 T_c2w：描述相机在世界中的位置和朝向
world-to-camera 外参 T_w2c = inverse(T_c2w)

Pixal3D 主要用 MoGe 估计内参/FOV，用于把图像特征对齐到 3D token。它不依赖完整相机位姿。

UV 反投则必须有完整相机几何：内参、外参、mesh UV 和可见性判断。只有内参时，无法判断图像中的像素对应 mesh 的哪一面。

如果 multi-view 图来自 diffusion，生成时要明确投影类型、视角、scale/FOV、target、分辨率和坐标系约定；反投前最好再做 pose refinement。

nvdiffrast 不替用户定义相机。相机约定需要先被转换成 Model/View/Projection 矩阵，最终通过 pos_clip = MVP @ vertices_homo 进入 rasterization。

模型架构速览

VibeVoice 本质上是 Qwen2.5 LLM + 扩散解码头 的混合架构，将大语言模型的文本理解能力与扩散模型的声学生成能力结合起来。

组件 作用 参数量
language_model (Qwen2.5) 自回归生成条件向量 ~1.5B
acoustic_tokenizer VAE 编解码音频波形 ~60M
semantic_tokenizer VQ-VAE 编码语义特征（自监督语音模型） ~60M
acoustic_connector 投影 acoustic latent → LLM hidden dim ~8M
semantic_connector 投影 semantic latent → LLM hidden dim ~8M
prediction_head（扩散） 噪声预测器，将 LLM hidden state → 音频 latent ~123M

文本与参考语音的融合机制

模型接受文本和参考音频两路输入，其融合发生在 embedding 层：

  1. 构造 prompt：系统提示 + [SPEECH_PLACEHOLDER × N] + Speaker 1: 目标文本 + <speech_start>
  2. Tokenize：整个 prompt 转为 token IDs
  3. 查 Embedding 表：每个 token ID 映射为向量
  4. 替换占位符：用参考音频的真实 embedding（acoustic_connector + semantic_connector 双路）覆盖占位符位置
inputs_embeds = embed_layer(input_ids).clone()
ref_speech_embeds = acoustic_connector(scaled_ac) + semantic_connector(se_latents)
ref_positions = speech_input_mask[0].nonzero()
inputs_embeds[0, ref_positions[:n_ref]] = ref_speech_embeds[0, :n_ref]

最终送入 LLM 的是 文本 embedding + 参考语音 embedding 混合序列——文本告诉它「说什么」，参考语音告诉它「用谁的声音说」。

推理中的 Semantic Feedback

自回归生成循环

每次生成一个音频 token 的流程：

for 每个输出 token:
    1. 扩散采样（DPM solver, ~10步）
       输入: last_hidden（LLM 条件向量）
       输出: scaled_latent [1, vae_dim]

    2. 反馈路由 → Acoustic-only 或 Acoustic+Semantic

    3. LLM 前向传播（单步，利用 KV cache）
       输出: 新的 last_hidden → 用于下一次扩散采样

两种反馈模式

模式 反馈内容 公式 速度
Acoustic-only 仅 acoustic embedding next = acoustic_connector(latent) 快
Acoustic+Semantic 解码→重编码→双路 next = ac_connector(latent) + se_connector(semantic_re_encode(decode(latent))) 慢（每步解码+重编码）

[!tip] 为什么 Semantic Feedback 是必要条件？
训练时 forward pass 用的是 acoustic + semantic 双路输入（有 ground truth semantic token）。Base model 推理默认用 acoustic-only——速度快，且 base model 没有严重的训练/推理分布偏移。

但微调后 prediction_head 发生变化，生成的 latent 分布与训练时不完全一致。若只反馈 acoustic，缺少 semantic 信号会让 LLM 偏离正常分布，导致退化（循环、噪声）。加上 semantic feedback 后，反馈信号与训练时一致，推理才稳定。

注意：这里的 semantic 不是 NLP 意义上「从文本得到的语义」，而是从音频波形中提取的语音内容特征（类似 HuBERT/w2vBERT）。因此必须先 decode → waveform → 再用 semantic_tokenizer 重新编码，无法从 acoustic latent 直接转换。

微调方案

方案一：Diffusion-only（主要方案）

冻结所有参数，仅训练 prediction_head（~123M params）。

原理：LLM hidden states 与 base model 完全一致，仅让扩散去噪器学习目标说话人的声学特征。

finetuning-tts/finetune_diffusion_only.py   # 训练
finetuning-tts/inference_diffusion.py       # 推理（含 semantic feedback）

方案二：LoRA on LLM（对比方案）

在 LLM 上应用 LoRA adapter（q/k/v/o/gate/up/down_proj）+ 同时训练 prediction_head。

LoRA 修改 LLM hidden states 使其更适应目标说话人，prediction_head 联合训练以适应修改后的 hidden states。

微调问题汇总

1.过拟合丢失生成EOS token能力

数据中 EOS：

# codec channel 中，EOS 只出现一次：在所有 audio codec 之后
input_ids[i, 8+text_ids_len-1+codec_ids_len, 1] = codec_eos_token_id

label中:

codec_0_labels[i, 8+text_ids_len-1+codec_ids_len] = codec_eos_token_id

每条训练样本中，EOS token只出现1次，而audio codec token出现几百次。
假设一条16秒音频在12HZ生成192 个codec frame, 模型看到的label分布是:

  • 192个正常codec token(如id=0~4095的各种值)
  • 1个EOS token(id=4198)

当模型在752条数据上训练20epoch出现过拟合：

  • 模型对每条样本的codec 序列背的滚瓜烂熟，loss趋于0
  • 在这种情况下，模型学习到的codec 序列和EOS绑定在一起，一起背住了
  • 当模型推理，输入没见过的text,模型找不到匹配的记忆模型，找不到EOS结束位置。

2.过拟合导致文本提示失效

loss如下:

outputs = model.talker(
    inputs_embeds=input_embeddings,  # text embedding + codec embedding 混合
    attention_mask=attention_mask,
    labels=codec_0_labels,           # 只监督 codec_0 的预测
    output_hidden_states=True
)
loss = outputs.loss + 0.3 * sub_talker_loss

loss只衡量"给定前面所有token, 预测下一个codec token的准确度"。现在考虑两种降低loss的策略:

  1. (理想泛化学习)理解文本内容 -> 根据文本语义发音 -> 预测对应的codec token
  2. (死记硬背)根据codec token序列[a,b,c,…] -> 直接记住a后面是b,b后面是c,预测下一个codec token

两种策略都能然loss下降，但策略2更简单，梯度下降优先选择这条路。

  1. 输入中 codec embedding 和 text embedding 是叠加的
input_embeddings = input_text_embedding + input_codec_embedding

模型可以选择性关注codec通道而忽略text通道
2. codec自身具有极强的自相关性 - 语音信号是连续的，相邻帧的codec是高度相似的。仅靠前几帧就能预测下一帧，不需要看文本。

3.过拟合导致重复循环

训练时：不存在重复循环
训练时用的是teacher forcing， 上下文中每一个codec token都是ground truth.所以训练时模型永远看到正确的上下文，不会出现循环。Loss 下降是正常的。
推理时
推理用的是autoregressive decoding - 每一步的输入是模型上一步生成的token.
这就是 train-inference mismatch（训练-推理不一致）问题,推理时很快偏离真实分布。

过拟合的模型在推理时会这样崩溃：

  1. 第 1 步：输入新文本，模型没见过，输出一个不太对的 codec token
  2. 第 2 步：输入 X，这个 X 不在任何训练样本的正常上下文中，模型又输出一个偏离的 Y
  3. 误差累积：几步之后，模型进入一个它在训练中完全没见过的状态空间
  4. 在陌生状态下：模型的输出概率分布变得混乱，很容易产生一个短序列 [A, B, C] 使得 A→B, B→C, C→A 形成闭环
  5. 闭环一旦形成就无法逃脱 — 因为模型没有"识别自己在循环"的能力，且不会生成 EOS

历史帧的注入机制：Concat to Token Sequence，而非 KV Cache

源码:transformer_helios_diffusers.py，历史帧不是通过 KV Cache 注入，而是通过将历史 latent 编码成 token，直接 concat 到当前 chunk 的 token sequence 前面，一起参与 Self-Attention。

核心机制：Token Concat Before Self-Attention

在 HeliosTransformer3DModel.forward() 中，流程如下：

Step 1：当前 chunk noisy latent patchify

hidden_states = self.patch_embedding(hidden_states)  
# shape: (B, inner_dim, T_cur, H', W')
hidden_states = hidden_states.flatten(2).transpose(1, 2)
# shape: (B, T_cur*H'*W', inner_dim)
original_context_length = hidden_states.shape[1]  # 记录当前chunk的token数

Step 2：Short 历史 latent 用独立 conv 编码后 concat 到前面

latents_history_short = self.patch_short(latents_history_short)  # 1x patch
# flatten → (B, T_short*H'*W', inner_dim)
hidden_states = torch.cat([latents_history_short, hidden_states], dim=1)
rotary_emb = torch.cat([rotary_emb_history_short, rotary_emb], dim=1)

Step 3：Mid 历史 latent 用2x 下采样 conv 编码后 concat

latents_history_mid = self.patch_mid(latents_history_mid)  # 2x patch（token数变少）
hidden_states = torch.cat([latents_history_mid, hidden_states], dim=1)

Step 4：Long 历史 latent 用4x 下采样 conv 编码后 concat

latents_history_long = self.patch_long(latents_history_long)  # 4x patch（token数更少）
hidden_states = torch.cat([latents_history_long, hidden_states], dim=1)

最终进入 Transformer Block 的 token 序列结构：

[ long_history_tokens | mid_history_tokens | short_history_tokens | current_chunk_tokens ]
                           ↑ 全部参与 Self-Attention ↑

与 KV Cache 的本质区别

特性 KV Cache（如 LLM） Helios 的做法
历史信息存储 存储历史 K/V，复用 存储历史 latent，每次重新编码
注入方式 历史 K/V 与当前 Q 做 attention 历史 token 与当前 token concat 后共同做 Self-Attention
Q 的范围 只有当前 token 做 Q 历史+当前 token 全部做 Q（但 output 只取 current 部分）
信息流向 单向：历史→当前 双向全 attention（但历史帧用 t=0 timestep）

两个精妙的设计细节

1. original_context_length：output 只取当前 chunk 部分

# HeliosOutputNorm.forward()
hidden_states = hidden_states[:, -original_context_length:, :]  # 只取当前 chunk 的 token 输出

历史 token 参与 attention 贡献 context，但最终 output 只对当前 chunk 的 token 解码，历史 token 的 output 被丢弃。

2. 历史帧使用 t=0 的 timestep embedding（已经干净的帧）

if self.zero_history_timestep:
    timestep_t0 = torch.zeros((1), dtype=timestep.dtype, ...)
    temb_t0, timestep_proj_t0, _ = self.condition_embedder(timestep_t0, ...)
    # 历史 token 使用 t=0 的 timestep embedding
    temb = torch.cat([temb_t0, temb], dim=1)

这在语义上非常合理：历史帧是已经生成好的"干净"帧，不需要去噪，所以 timestep=0。

3. 多尺度历史：空间&时间分辨率递减

short_history: patch_size = (1,2,2)  → 原始分辨率，距当前最近 1帧
mid_history:   patch_size = (2,4,4)  → 空间 1/2，时间 1/2，中期 2帧
long_history:  patch_size = (4,8,8)  → 空间 1/4，时间 1/4，远期 16帧

时间越久的历史，其 token 数越少（分辨率越低），这样既保留了长程依赖，又控制了 sequence length 的增长—— 类似于人类记忆的多分辨率压缩。

总结

Helios 历史帧的注入方式是：Multi-scale Token Concatenation（多尺度历史 token 拼接）到 Self-Attention 的 token 序列前。

  • ✅ 不是 KV Cache（没有缓存 K/V，每次重新编码历史 latent）。
  • ✅ 不是替换 T 中的部分帧（当前 chunk 的 T 维度是完整的 noisy latent，历史从 token 维度拼接进来）。
  • ✅ Self-Attention 可以看到历史 token（全局注意力），但不消耗于 output。

一、 摘要与核心贡献

本文对扩散模型蒸馏（Diffusion Model Distillation, DMD）的内在机制提出了颠覆性的见解。传统的 DMD 算法被认为是通过匹配学生模型与教师模型的输出分布来工作的。然而，本文作者通过对 DMD 训练目标的严格分解发现：

  1. 真正的引擎并非分布匹配：在文本生成图像等复杂任务中，驱动“少步生成”的核心动力并非分布匹配，而是被长期忽视的 CFG 增强 (CFG Augmentation, CA)。

  2. 分布匹配的角色是正则项：分布匹配（Distribution Matching, DM）项实际上起到的是“正则化”作用，用于确保训练稳定性并消除图像伪影。

  3. 解耦与改进：基于这种分工解耦，作者提出了解耦的重噪调度（Decoupled Re-noising Schedules），显著提升了蒸馏效果。

二、 DMD 的重新审视与数学分解

2.1 理论回顾

DMD 旨在训练一个学生生成器 GθG_{\theta}Gθ​，通过最小化 积分 KL 散度 (IKL) 来模拟教师模型的输出分布：

LIKL(preal,pfake)=∫01KL(preal,τ∣∣pfake,τ)dτ(1)\mathcal{L}_{IKL}(p_{real}, p_{fake}) = \int_{0}^{1} \mathbb{KL}(p_{real, \tau} || p_{fake, \tau}) d\tau \tag{1} LIKL​(preal​,pfake​)=∫01​KL(preal,τ​∣∣pfake,τ​)dτ(1)

理想情况下，参数 θ\thetaθ 的梯度估计为：

∇θLtheory=E[−(scondreal(xτ)−scondfake(xτ))∂Gθ(zt)∂θ](2)\nabla_{\theta} \mathcal{L}_{theory} = \mathbb{E} \left[ - \left(s_{cond}^{real}(\mathbf{x}_{\tau}) - s_{cond}^{fake}(\mathbf{x}_{\tau})\right) \frac{\partial G_{\theta}(z_t)}{\partial \theta} \right] \tag{2} ∇θ​Ltheory​=E[−(scondreal​(xτ​)−scondfake​(xτ​))∂θ∂Gθ​(zt​)​](2)

2.2 实践中的偏离

在实际的大规模文本图像任务中，开发者发现直接使用公式 (2) 效果极差。为了提升效果，实践中会对教师分数采用 分类器自由引导 (CFG)：

∇θLDMD=E[−(scfgreal(xτ)−scondfake(xτ))∂Gθ(zt)∂θ](3)\nabla_{\theta} \mathcal{L}_{DMD} = \mathbb{E} \left[ - \left(s_{cfg}^{real}(\mathbf{x}_{\tau}) - s_{cond}^{fake}(\mathbf{x}_{\tau})\right) \frac{\partial G_{\theta}(z_t)}{\partial \theta} \right] \tag{3} ∇θ​LDMD​=E[−(scfgreal​(xτ​)−scondfake​(xτ​))∂θ∂Gθ​(zt​)​](3)

其中，scfgreals_{cfg}^{real}scfgreal​ 的定义为：

scfgreal(xτ)=suncondreal(xτ)+α(scondreal(xτ)−suncondreal(xτ))(4)s_{cfg}^{real}(\mathbf{x}_{\tau}) = s_{uncond}^{real}(\mathbf{x}_{\tau}) + \alpha (s_{cond}^{real}(\mathbf{x}_{\tau}) - s_{uncond}^{real}(\mathbf{x}_{\tau})) \tag{4} scfgreal​(xτ​)=suncondreal​(xτ​)+α(scondreal​(xτ​)−suncondreal​(xτ​))(4)

α\alphaα 为引导尺度。这种理论与实践的差异一直没有得到系统的科学解释。

2.3 核心分解公式

作者通过代入公式 (4) 到公式 (3)，推导出了以下精妙的解耦形式：

∇θLDMD=E[−((scondreal−scondfake)⏟Δreal−fake(DM)+(α−1)(scondreal−suncondreal)⏟Δcfgreal(CA))∂Gθ(zt)∂θ](6)\nabla_{\theta} \mathcal{L}_{DMD} = \mathbb{E} \left[ - (\underbrace{(s_{cond}^{real} - s_{cond}^{fake})}_{\Delta^{real-fake} (\text{DM})} + \underbrace{(\alpha - 1)(s_{cond}^{real} - s_{uncond}^{real})}_{\Delta_{cfg}^{real} (\text{CA})}) \frac{\partial G_{\theta}(z_t)}{\partial \theta} \right] \tag{6} ∇θ​LDMD​=E⎣⎢⎢⎡​−(Δreal−fake(DM)(scondreal​−scondfake​)​​+Δcfgreal​(CA)(α−1)(scondreal​−suncondreal​)​​)∂θ∂Gθ​(zt​)​⎦⎥⎥⎤​(6)

由此可见，DMD 的更新梯度由两部分组成：

  1. 分布匹配项 (DM)：对应理论推导，致力于减小学生与教师的分布差异。

  2. CFG 增强项 (CA)：直接将增强的引导信号转化为生成器的更新方向。

三、 机制分析：矛与盾的分工

3.1 实验验证：谁在驱动转换？

作者通过消融实验对比了仅使用 CA、仅使用 DM 以及全目标（DMD）的效果。

图 2：CA 能够驱动有效的少步转换，但单独使用会导致训练崩溃（出现过饱和和噪声）；DM 能保持稳定，但单独使用的能力转换效率较低。

结论 1 (CA 是引擎)：生成器产生高质量少步样本的能力几乎完全归功于 CA 项的“强力驱动”。

结论 2 (DM 是正则项)：DM 项防止了训练过程的发散，通过纠偏确保了最终输出的质量。

四、 原理深度解析

4.1 CA 引擎的工作原理：刻录决策路径

本文提出了一个深刻的洞察：扩散模型是序列生成器。

  • 低频到高频：模型先定构图（如：是猫还是狗），再定细节。

  • CFG 作为确定性干预：CFG 实际上提供了一个确定性的决策偏置。通过 CA 项，这种偏置被“刻录”（Bake）到了生成器的参数中。

  • 类比预测：就像预测“Elon”后面必然跟“Musk”一样，CA 强制模型学习了从噪声到特定高质量内容的捷径，从而跳过了原本多步采样的中间决策过程。

4.2 DM 的纠错机制

DM 是如何抑制伪影的？

作者设计了观察实验：单独用 CA 训练生成器，但让一个辅助模型（Fake model）冷眼旁观。

图 4：(a) 展示了不同重噪水平 τ\tauτ 对 CA 的影响——高燥水平控构图，低燥水平控细节；(b) 展示了当 CA 产生棋盘格伪影时，教师模型（真实分数）没有该错误，而辅助模型（模拟分布）学习到了该错误。两者相减产生的负梯度有效抵消了伪影。

五、 改进：解耦的重噪调度 (Decoupled Schedule)

基于以上分析，作者提出了针对性的改进方案。既然 CA 是针对性引擎，而 DM 是全局正则项，它们不应该被迫分享相同的重噪时间戳选择。

作者提出了解耦的 DMD (d-DMD) 梯度公式：

∇θLd−DMD=E[−(ΔDM(τDM)+ΔCA(τCA))∂Gθ∂θ](8)\nabla_{\theta} \mathcal{L}_{d-DMD} = \mathbb{E} \left[ - (\Delta_{DM}(\tau_{DM}) + \Delta_{CA}(\tau_{CA})) \frac{\partial G_{\theta}}{\partial \theta} \right] \tag{8} ∇θ​Ld−DMD​=E[−(ΔDM​(τDM​)+ΔCA​(τCA​))∂θ∂Gθ​​](8)

推荐配置 (Decoupled-Hybrid)：

  • CA 专注细节 (τCA>t\tau_{CA} > tτCA​>t)：只在生成器当前阶段还没解决的噪声水平应用 CA 引导。

  • DM 全局监管 (τDM∈[0,1]\tau_{DM} \in [0, 1]τDM​∈[0,1])：始终保持全范围的分布监控。

图 5：解耦混合策略（底部）生成的图像细节最丰富，且没有过度饱和（“油腻感”）的问题。

六、 总结与局限性

总结：

本文通过对 DMD 的解耦分析，揭示了“CFG 增强”才是扩散生成器步数压缩的核心引擎。这一发现将分布式匹配回归到其本职工作——正则化。这种视角的转变不仅提供了更强的理论解释性，也直接催生了更高效的训练策略。

局限性：

尽管本文对“为什么 CA 有效”给出了精彩且具有前瞻性的类比（参考 A 章节），但其底层数学原理仍有待进一步深入探讨。特别是 CFG 本身的机制在学术界仍有一定神秘感。
