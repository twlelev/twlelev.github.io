0.使用前配置

/plugins 在Manage plugins中的Marketplaces中添加https://github.com/anthropics/claude-plugins-official.git，随后便能在Plugins中添加插件，一般排名前几的都是比较好用的。

1. CLAUDE.md — 最被低估的功能

如果你只记住这篇文章的一件事，那就是：写好 claude.md。

CLAUDE.md 是 Claude Code 每次对话启动时自动读取的指令文件，就像给新同事的入职文档。/init 可以让claude自动总结仓库并编写CLAUDE.md。
好的 CLAUDE.md 应该包含什么

  • 写 Claude 从代码里读不出来的东西——"为什么"比"是什么"更重要
  • 保持在 200 行以内（太长会导致 Claude 忽略规则）
  • 不要放频繁变化的内容，用链接代替

四级层级配置
Claude Code 支持多级 CLAUDE.md，按优先级加载：全局 → 用户 → 工作区 → 文件夹级别
示例内容

  • 项目名称和常用命令
  • 编码规范
  • 架构决策背景
  • 常见陷阱

2. .claude/rules/ 目录 — 进阶规则管理

在 .claude/rules/ 目录下创建带 YAML frontmatter 的规则文件，可以针对特定文件类型生效。规则仅在访问匹配文件时加载，节省上下文空间。

3. 指令质量决定输出质量

差指令 vs 好指令

差的指令 好的指令
“给用户列表加搜索功能” “在 @src/pages/UserList.tsx 的表格上方加搜索框。调用后端 API /api/users?search=xxx，不要前端过滤。使用 300ms 防抖。样式与 Filter 组件保持一致。”

好指令的三要素：目标 + 约束 + 上下文

官方推荐技巧

  • 用 @filepath 直接引用具体文件
  • 遇到 UI 问题直接贴截图
  • 提供验证标准（测试用例、预期输出）
  • 明确说明**“不要做什么”**

什么时候可以用模糊指令

  • 探索性任务
  • 理解架构
  • 代码改进建议

4. Plan Mode — 先想后做

Plan Mode 下 Claude Code 只读取文件和分析，不做任何修改，生成详细的实现计划。
这里可以通过安装**superpowers**插件实施。

安装后每次执行任务时看到的 spec → plan → todo 流程，是这三个skill协作的结果：

  1. superpowers:brainstorming — 负责探索需求和写spec。在任何创建/修改功能前触发，和你讨论意图、约束、设计方案。
  2. superpowers:writing-plans — 负责把spec转化为实施计划。有了明确需求后，生成分步骤的implementation plan。
  3. superpowers:executing-plans / superpowers:subagent-driven-development — 负责执行计划并更新TodoWrite。执行时把计划拆成todo items，逐步完成并标记。

5. 子 Agent — AI 并行处理

子 Agent 在独立的上下文窗口中处理任务。
内置类型

类型 能力 用途

| Explore | 只读，快速代码搜索 | 查找代码、理解结构 |
| Plan | 只读分析 | 制定方案 |
| General | 完整功能 | 复杂独立任务 |

核心价值
隔离上下文——测试输出、日志、搜索结果留在子 Agent 的上下文中，只返回摘要给主对话，避免污染主上下文。

自定义 Agent
在 .claude/agents/ 目录下用 YAML 配置创建。

---
name: code-reviewer
description: 代码审查专家。代码改动后自动触发。
tools: Read, Grep, Glob, Bash
model: sonnet
---

你是一位资深代码审查者。检查以下维度：
- 代码清晰度和可读性
- 错误处理是否完整
- 有没有暴露敏感信息
- 输入验证
- 性能隐患

然后在对话中用 @“code-reviewer (agent)” 调用它。

6. 上下文管理 — 关键但容易忽略

什么会占用上下文

  • 对话历史
  • Claude 读取的文件内容
  • 命令输出
  • CLAUDE.md（每次都加载）
  • Memory 文件（前 200 行）
  • Skill 和 MCP 工具定义

五个实用策略

1) /clear — 清空对话

清空对话但保留 CLAUDE.md。完成一项任务就清理。

2) /compact — 手动压缩

可以带焦点参数：/compact focus on API changes and test results

3) /context — 查看上下文用量

按文件和工具显示上下文使用明细。

4) 子 Agent 隔离噪音

把运行测试、分析代码等任务放到子 Agent 中，避免大量输出占满主上下文。

5) CLAUDE.md 中加压缩指令

告诉 Claude 在压缩时应该保留哪些信息。
Memory vs CLAUDE.md

Memory 存储 Memory 不应存储
个人偏好 代码细节
项目约定 Git 历史
历史决策 临时状态

用 /memory 命令管理已加载的 Memory 文件。

7. 权限模式 — 五个级别

通过 Shift+Tab 循环切换权限模式，从无限制到只读（Plan Mode）逐级递增限制。

在 .claude/settings.json 中配置权限


规则优先级：deny → ask → allow（deny 优先级最高）


{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Edit(src/**/*.ts)",
      "Read(*.md)",
      "Glob",
      "Grep",
      "WebFetch",
      "WebSearch",
      "Agent",
      "TodoWrite",
      "mcp__plugin_context7_context7__query-docs",
      "mcp__plugin_context7_context7__resolve-library-id"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Edit(.env)"
    ]
  }
}

配置文件层级

文件 用途
.claude/settings.json 团队共享
~/.claude/settings.json 个人配置
.claude/settings.local.json 敏感配置，需加入 .gitignore

8. Hooks — 让规则变成铁律

CLAUDE.md 是建议；Hooks 是无条件执行的规则。
关键事件类型

事件 触发时机
PreToolUse 工具使用前
PostToolUse 工具使用后
SessionStart 会话开始时
Compact 上下文压缩时

实际用例

  • 编辑后自动格式化（Prettier hook）
  • 阻止修改生产环境配置
  • 压缩后重新注入关键信息

Hook 命令类型

类型 说明
command Shell 命令
http POST 请求
prompt yes/no 决策
agent 子 Agent 验证

退出码： 0 = 允许，2 = 阻止

9. Git 工作流 — Worktree 安全网

Git Worktree
在 .claude/worktrees/ 中创建隔离的分支副本。Claude 的编辑无风险，完成后自动清理或提示是否保留。

Git 使用规则

  1. 永远不要自动 push（Claude 只能 commit）
  2. 频繁 commit，保持 /rewind 回滚能力
  3. 对危险操作设 deny 权限（force push、reset 等）
  4. 用 --from-pr 参数加载 PR 上下文做代码审查

10. 快捷键与命令速查

最常用斜杠命令

命令 功能

11. Extended Thinking — 深度推理模式

通过 Option+T（Mac）或 Alt+T（Windows/Linux）开启。
适合使用的场景

  • 复杂架构决策
  • 多因素调试
  • 重构策略选择
  • 方案对比分析

不需要使用的场景

  • 简单代码修改
  • 格式化/重命名
  • 明显的 bug 修复

在 prompt 中提到 “ultrathink” 可以强制开启最大推理深度。

12. MCP Server — 外部工具集成

添加方式


claude mcp add [name] -- [command]

配置文件
在 .mcp.json 中配置。

注意事项

每个 MCP 服务器会增加 100-500+ tokens 的工具定义开销。用 /mcp 命令检查开销。建议最多 2-3 个核心服务器。
