模型架构速览

VibeVoice 本质上是 Qwen2.5 LLM + 扩散解码头 的混合架构，将大语言模型的文本理解能力与扩散模型的声学生成能力结合起来。

组件 作用 参数量
language_model (Qwen2.5) 自回归生成条件向量 ~1.5B
acoustic_tokenizer VAE 编解码音频波形 ~60M
semantic_tokenizer VQ-VAE 编码语义特征（自监督语音模型） ~60M
acoustic_connector 投影 acoustic latent → LLM hidden dim ~8M
semantic_connector 投影 semantic latent → LLM hidden dim ~8M
prediction_head（扩散） 噪声预测器，将 LLM hidden state → 音频 latent ~123M

文本与参考语音的融合机制

模型接受文本和参考音频两路输入，其融合发生在 embedding 层：

  1. 构造 prompt：系统提示 + [SPEECH_PLACEHOLDER × N] + Speaker 1: 目标文本 + <speech_start>
  2. Tokenize：整个 prompt 转为 token IDs
  3. 查 Embedding 表：每个 token ID 映射为向量
  4. 替换占位符：用参考音频的真实 embedding（acoustic_connector + semantic_connector 双路）覆盖占位符位置
inputs_embeds = embed_layer(input_ids).clone()
ref_speech_embeds = acoustic_connector(scaled_ac) + semantic_connector(se_latents)
ref_positions = speech_input_mask[0].nonzero()
inputs_embeds[0, ref_positions[:n_ref]] = ref_speech_embeds[0, :n_ref]

最终送入 LLM 的是 文本 embedding + 参考语音 embedding 混合序列——文本告诉它「说什么」，参考语音告诉它「用谁的声音说」。

推理中的 Semantic Feedback

自回归生成循环

每次生成一个音频 token 的流程：

for 每个输出 token:
    1. 扩散采样（DPM solver, ~10步）
       输入: last_hidden（LLM 条件向量）
       输出: scaled_latent [1, vae_dim]

    2. 反馈路由 → Acoustic-only 或 Acoustic+Semantic

    3. LLM 前向传播（单步，利用 KV cache）
       输出: 新的 last_hidden → 用于下一次扩散采样

两种反馈模式

模式 反馈内容 公式 速度
Acoustic-only 仅 acoustic embedding next = acoustic_connector(latent) 快
Acoustic+Semantic 解码→重编码→双路 next = ac_connector(latent) + se_connector(semantic_re_encode(decode(latent))) 慢（每步解码+重编码）

[!tip] 为什么 Semantic Feedback 是必要条件？
训练时 forward pass 用的是 acoustic + semantic 双路输入（有 ground truth semantic token）。Base model 推理默认用 acoustic-only——速度快，且 base model 没有严重的训练/推理分布偏移。

但微调后 prediction_head 发生变化，生成的 latent 分布与训练时不完全一致。若只反馈 acoustic，缺少 semantic 信号会让 LLM 偏离正常分布，导致退化（循环、噪声）。加上 semantic feedback 后，反馈信号与训练时一致，推理才稳定。

注意：这里的 semantic 不是 NLP 意义上「从文本得到的语义」，而是从音频波形中提取的语音内容特征（类似 HuBERT/w2vBERT）。因此必须先 decode → waveform → 再用 semantic_tokenizer 重新编码，无法从 acoustic latent 直接转换。

微调方案

方案一：Diffusion-only（主要方案）

冻结所有参数，仅训练 prediction_head（~123M params）。

原理：LLM hidden states 与 base model 完全一致，仅让扩散去噪器学习目标说话人的声学特征。

finetuning-tts/finetune_diffusion_only.py   # 训练
finetuning-tts/inference_diffusion.py       # 推理（含 semantic feedback）

方案二：LoRA on LLM（对比方案）

在 LLM 上应用 LoRA adapter（q/k/v/o/gate/up/down_proj）+ 同时训练 prediction_head。

LoRA 修改 LLM hidden states 使其更适应目标说话人，prediction_head 联合训练以适应修改后的 hidden states。

实验设计

固定参数：batch_size=1，gradient_accumulation=8（有效 batch=8），lr_scheduler=cosine，warmup_ratio=0.05，weight_decay=0.01，bf16 精度，训练数据 4704 样本。

搜索变量：

变量 搜索范围
微调方法 diffusion-only / LoRA
学习率 (lr) 1e-7 ~ 5e-6
训练轮数 (epochs) 1 ~ 20
LoRA rank 8, 16
LoRA alpha 16, 32

评估指标：

  • 说话人相似度：Resemblyzer VoiceEncoder，生成音频与参考音频的余弦相似度
  • ASR 正确性：faster-whisper (medium)，检查生成内容是否正确

学习率敏感度：极窄有效窗口

prediction_head 对学习率极其敏感：

lr 范围 效果
5e-7 最优：ASR 正确，相似度最高
7.5e-7 可用但偏弱
2e-7 ~ 2.5e-7 安全但偏弱
1e-6 危险：ASR 开始退化（循环、重复）
≥ 2e-6 完全损坏：输出噪声

有效学习率窗口：2e-7 ~ 5e-7，最优点在 5e-7。

LoRA 方案分析

lr 结果
1e-4 ~ 5e-6 完全损坏（sim 0.45-0.58，ASR 错误）
1e-7 首次可用（sim 0.894，ASR 正确）

LoRA 在极低 lr 下才可工作，但收益不如 diffusion-only（0.894 vs 0.923）。LLM hidden states 对 LoRA 修改非常敏感——这也从反面印证了「冻结 LLM，只调扩散头」的正确性。

关键结论

[!success] 最优配置

  • 方法：Diffusion-only（仅训练 prediction_head，~123M）
  • 学习率：5e-7
  • 训练轮数：10 epochs
  • 调度器：cosine，warmup_ratio=0.05
  • 推理：semantic feedback + CFG 1.3 + DPM 10 steps
  • 结果：说话人相似度 0.9234（base model 0.829，+11.3%）
  1. Diffusion-only > LoRA：冻结 LLM 只调 prediction_head 是最优策略。修改 LLM hidden states 会严重破坏语音生成质量。
  2. lr 窗口极窄：有效范围仅 2e-7~5e-7。≥1e-6 即 ASR 退化，≥2e-6 完全损坏。
  3. 10 epochs 恰好：少训欠拟合，多训过拟合，曲线对称。
  4. Semantic feedback 必须开启：微调后推理必须反馈 acoustic + semantic 双路，否则分布偏移导致循环/噪声。
  5. 目标 0.95 未达成：Diffusion-only 的天花板约为 0.92-0.93。进一步突破可能需要更多训练数据或探索数据增强策略。

最佳模型与推理命令

experiments/E06/model/prediction_head.pt

python finetuning-tts/inference_diffusion.py \
    --base_model microsoft/VibeVoice-1.5B \
    --prediction_head_path ./experiments/E06/model/prediction_head.pt \
    --ref_audio ./ref.ogg \
    --text "要合成的文本" \
    --output_path ./output.wav \
    --cfg_scale 1.3 \
    --num_inference_steps 10

实验环境：4× NVIDIA RTX 4090 (24GB)，2 GPU 并行训练，总运行时间约 12 小时（14 个实验，7 轮并行）。
