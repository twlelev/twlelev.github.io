核心：屏幕空间稠密先验(screen-space dense priors)。

“search uv” 机制详解

“search uv” 的本质是:用 KNN 在网络预测的 UV 图里,为每个 FLAME 顶点反查出它对应的图像像素位置,从而把"per-pixel 的 UV 预测"转成"per-vertex 的 2D 重投影目标"来拟合 FLAME 参数。

个人总结，通俗解释：
网络预测uv coordinates, 得到屏幕像素空间与uv坐标的映射，因为canonical uv坐标与3d顶点存在对应的关系，因此通过转换可以得到屏幕像素空间与3d顶点的对应。
预测的uv坐标不一定准确，所以通过search_uv优化，将其与canonical uv比较(最邻近搜索，排除距离远的奇异点等)得到屏幕空间每一个像素对应的3d坐标作为gt来监督。

A.1 前提:两套 UV 的定义

(a) Canonical UV(模板顶点的固定 UV) — losses.py:60-61

self.can_uv = torch.from_numpy(np.load(env_paths.FLAME_UV_COORDS)[self.valid_verts, :]).cuda().unsqueeze(0).float()
self.can_uv[..., 1] = (self.can_uv[..., 1] * -1) + 1   # OBJ-v 翻成 image-style v
  • FLAME_UV_COORDS 是 FLAME 模板每个顶点在 UV 图集上的固定坐标,与姿态/表情无关(对应正文 §5 的"横向独立平面 UV ∈ [0,1]²")。
  • valid_verts(losses.py:56-59)剔除后脑勺/脖子等不可靠区域;stricter_mask 时用更窄的集合。
  • 第 61 行 v = 1 - v,把 OBJ 约定翻成图像坐标,与网络预测 UV 同约定。

(b) Predicted UV(网络对每个图像像素的 UV 预测) — 即 uv_map,网络输出 2 通道,每像素值是"该像素落在 FLAME UV 图集的哪个位置"。

A.2 核心搜索:compute_corresp

losses.py:79-95:


gt_uv = gt[:, :2, :, :].permute(0, 2, 3, 1)           # 取预测 UV 的 2 通道
gt_uv = gt_uv.reshape(gt_uv.shape[0], -1, 2)          # B x (H*W) x 2,每个像素当成 UV 空间一个点
can_uv = self.can_uv.repeat(gt_uv.shape[0], 1, 1)     # B x N_verts x 2
knn_result = knn_points(can_uv, gt_uv)                # 关键:在 UV 空间做最近邻
pixel_position_width  = knn_result.idx % gt.shape[-1] # 展平索引拆回 (x, y)
pixel_position_height = knn_result.idx // gt.shape[-2]
self.dists = knn_result.dists.clone()
self.gt_2_verts = torch.cat([pixel_position_width, pixel_position_height], dim=-1)

要点:

  • 搜索发生在 UV 空间,而非图像空间。 knn_points(can_uv, gt_uv) 对每个顶点的 canonical UV,在全部 H*W 个像素的预测 UV 里找欧氏距离最近的像素。含义:“哪个图像像素的预测 UV,最接近这个顶点本该所在的 UV?”
  • knn_result.idx 是展平像素下标,% width / // width 还原成 (x, y) → 存进 gt_2_verts(每个有效顶点一个 2D 目标点)。
  • dists 记录每个匹配的 UV 距离,用作置信度门控。
  • @torch.compiler.disable(losses.py:79):KNN 不走 torch.compile。
  • GT 侧 UV 在 uv_pred_to_mesh 里(uv.py:51-55)先 (x+1)/2 归一化、用 mask 清背景、把 abs>=1 的无效预测设成 10(推到 UV 无穷远), 保证无效像素永不被选中。

A.3 把搜索结果变成 loss:compute_loss

losses.py:99-127:

valid_proj_v = proj_vertices[:, self.valid_vertex_index, ..., :2]  # 当前 FLAME 参数投影出的顶点 2D
v_dist_2d = (gt_2_verts - valid_proj_v)                            # 目标像素 - 投影像素
uv_loss = (
    (v_dist_2d / self.gt.shape[-1])                               # 归一化到图像尺度
    * (self.dists < self.delta)                                   # 门控① UV 距离
    * (v_dist_2d.abs().sum(dim=-1) < self.dist_uv).unsqueeze(-1)  # 门控② 2D 位移
    * not_occluded.unsqueeze(-1)                                  # 门控③ 遮挡
).abs().mean()

l2_loss=True 时改用平方 loss ×100,否则 L1。
含义：让每个有效顶点的2D投影位置，去逼近它在图像里对应像素的位置

仓库代码有两种作用不同层级的loss:

  • loss/uv_pixel: 把FLAME mesh用光栅化器渲染成一张UV图ops['uv_images'],和网络预测的 uv_map 做逐像素比较。是render-and-compare、图像空间的稠密loss.
  • loss/uv(本文主角): 不渲染，直接在顶点级别用KNN对应做2D距离。

A.4 在 tracker 里的调用时序

tracker.py:1145-1151:

if self.uv_loss_fn.gt_2_verts is None:
    self.uv_loss_fn.compute_corresp(uv_map, selected_frames=selected_frames)  # search 只做一次
uv_loss = self.uv_loss_fn.compute_loss(proj_vertices, ..., is_visible_verts_idx=is_visible_verts_idx)
losses['loss/uv'] = uv_loss * self.config.uv_map_super
  • KNN 搜索在一帧/一个优化阶段内只跑一次(is_next() 在 tracker.py:1838 重置 gt_2_verts)。对应关系来自固定的预测 UV,不随 FLAME 参数迭代变化;变化的只是 proj_vertices,所以内循环只重算 compute_loss。昂贵的 KNN 不进优化内循环——类似 ICP 的"先定对应,再求变换"。
  • UVLoss 初始化(tracker.py:163-168)参数全来自 config,compute_loss 被 torch.compile 加速,compute_corresp 不被 compile。
  • finish_stage1(losses.py:70-74,tracker.py:1893):粗→细阶段切换时换更严格的 delta_uv_fine / dist_uv_fine。
